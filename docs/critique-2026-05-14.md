## 5. Sublimation 方向性

本批判の core insight: **既存の 4-commit work は doc 主体で実装より遥かに重い**。配布検証フェーズの前段としては正しいが、検証中の実コード feedback (P1 observability) が回り始めれば doc は再構築 (不要 doc 削除 / 統合) 圧力が出る。

具体的な昇華 path:

1. **P1-2 observability 着手で「実 user 行動データ」を集める → 6 月後半に doc cleanup を 1 session で実施**
   - 「実発生していない event」を doc から削除 (例: BS-1/BS-2/BS-5 が実発生しないなら deferred のまま eternal、削除候補)
   - 「実 user が混乱した点」を OPERATIONS.md §3 (緊急復旧) に追記
2. **critic vote の context 分割化**: 次回 critic vote は **commit per critic** で 3-critic を分散、context overflow を回避
3. **`docs/critique-*` を時系列で残す**: 本 file がその 1 本目。月次 critique を残して **検証フェーズ retrospective** に使う
